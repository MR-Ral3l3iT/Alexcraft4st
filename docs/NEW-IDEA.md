1) ภาพรวมแนวคิดของระบบเพิ่มเติม

ผู้ใช้ 1 คน จะมีสถานะหลัก ๆ ดังนี้
	1.	ยังไม่ลงทะเบียน
	2.	ลงทะเบียนแล้ว รอแอดมินคอนเฟิร์ม
	3.	คอนเฟิร์มแล้ว รอเช็คอิน
	4.	เช็คอินแล้ว เข้าสู่โหมดเล่นเกมเก็บเลเวล
	5.	เช็คเอาท์ / จบกิจกรรม

ทุกครั้งที่สถานะเปลี่ยน ระบบจะเปลี่ยน Rich Menu ของ user คนนั้น ให้เห็นเฉพาะปุ่มที่ควรกดในช่วงนั้น โดย per-user rich menu มีลำดับความสำคัญสูงกว่า default rich menu และผูก/ถอดได้ผ่าน Messaging API  ￼

⸻

2) Flow หลักของระบบ

Phase A: ก่อนวันงาน

สถานะ A1 — ยังไม่ลงทะเบียน

ผู้ใช้เห็น Rich Menu ชุดเริ่มต้น

เมนู
	•	ลงทะเบียน
	•	ตรวจสอบสถานะ
	•	สถานที่จัดงาน

การทำงาน
	1.	ผู้ใช้กด “ลงทะเบียน”
	2.	เปิด LIFF หน้า Register
	3.	LIFF ดึงข้อมูลพื้นฐานจาก LINE ได้ เช่น userId / โปรไฟล์พื้นฐาน
	4.	ผู้ใช้กรอกข้อมูลเพิ่ม เช่น ชื่อ เบอร์ จำนวนที่นั่ง
	5.	ระบบบันทึก booking เป็นสถานะ PENDING

LIFF เป็นเว็บแอปบน LINE ที่ใช้ข้อมูลผู้ใช้จากแพลตฟอร์ม LINE ได้ และต้อง liff.init() ทุกครั้งเมื่อเปิดหน้าใหม่ตามเอกสาร  ￼

⸻

สถานะ A2 — ลงทะเบียนแล้ว รอคอนเฟิร์ม

หลังลงทะเบียน ระบบเปลี่ยนเป็นเมนูรออนุมัติ

เมนู
	•	ตรวจสอบสถานะ
	•	สถานที่จัดงาน
	•	ติดต่อแอดมิน

การทำงาน
	1.	แอดมินเปิดหลังบ้าน
	2.	ตรวจสอบรายการลงทะเบียน
	3.	กด confirm หรือ cancel
	4.	ถ้า confirm:
	•	booking status = CONFIRMED
	•	ระบบส่งข้อความยืนยันใน LINE
	•	ระบบเปลี่ยน per-user rich menu เป็นชุด “พร้อมเช็คอิน”

⸻

Phase B: วันงาน — เช็คอิน

สถานะ B1 — คอนเฟิร์มแล้ว รอเช็คอิน

ผู้ใช้เห็นเมนูสำหรับหน้างาน

เมนู
	•	เช็คอิน
	•	สถานที่จัดงาน
	•	รายละเอียดงาน / กติกา

Flow เช็คอิน
	1.	ผู้ใช้กด “เช็คอิน”
	2.	เปิด LIFF หน้า Check-in
	3.	ระบบตรวจว่า:
	•	booking ต้องเป็น CONFIRMED
	•	ยังไม่เคยเช็คอิน
	•	อยู่ในช่วงเวลาที่กำหนด
	4.	ระบบขอ location จากผู้ใช้
	5.	LIFF / web app ใช้ Geolocation API เพื่ออ่านพิกัดปัจจุบัน
	6.	ระบบคำนวณระยะจากพิกัดงาน
	7.	ถ้าอยู่ในรัศมีที่กำหนด เช่น 80–100 เมตร → เช็คอินสำเร็จ
	8.	เปลี่ยนสถานะเป็น CHECKED_IN
	9.	บันทึกเวลาเช็คอิน
	10.	เปลี่ยน per-user rich menu เป็นชุด “เล่นเกมในงาน”

เอกสาร LINE ระบุว่า LINE MINI App/LIFF รองรับการใช้ Geolocation API ได้ และใช้ความสามารถ HTML5 ได้เกือบทั้งหมด  ￼

กติกาเช็คอินที่แนะนำ
	•	รัศมี 80–100 เมตร
	•	เปิดได้เฉพาะช่วงเวลางาน เช่น 17:30–20:00
	•	ถ้า location fail ให้มี fallback ให้ staff ช่วย check-in จากหลังบ้าน

⸻

Phase C: หลังเช็คอิน — เกมเก็บเลเวล

สถานะ C1 — เช็คอินแล้ว

เมื่อเช็คอินสำเร็จ ผู้ใช้เข้าสู่โหมดกิจกรรม

เมนู
	•	เติมเบียร์
	•	ระดับพลังของฉัน
	•	เช็คเอาท์

⸻

3) Flow เกมเก็บเลเวล

แนวคิดคือ
1 แก้ว = 1 แต้ม
ครบตามเกณฑ์ = เลเวลอัป

Level ที่แนะนำ

Level 1: Fresh Drinker
	•	1–2 แก้ว
	•	คาแรกเตอร์: มือใหม่สายชิลล์

Level 2: Buzz Explorer
	•	3–5 แก้ว
	•	คาแรกเตอร์: เริ่มเข้าฟีล

Level 3: Beer Master
	•	6+ แก้ว
	•	คาแรกเตอร์: ตัวตึงประจำงาน

⸻

Flow เติมเบียร์
	1.	ผู้ใช้กด “เติมเบียร์”
	2.	เปิด LIFF หน้า Beer Power
	3.	ระบบตรวจว่า:
	•	ต้อง check-in แล้ว
	•	ยังไม่ checkout
	4.	กดปุ่ม + เติมเบียร์
	5.	ระบบเพิ่ม beerCount + 1
	6.	ระบบคำนวณ level ใหม่
	7.	ถ้าเลเวลเปลี่ยน ให้แสดง popup Level Up

⸻

Logic กันกดมั่ว

เพราะเป็น free flow และไม่อยากให้หน้างานยุ่ง ผมแนะนำใช้แบบนี้

แบบที่เวิร์คสุด
	•	ให้ user กดเอง
	•	มี cooldown เช่น 3–5 นาทีต่อ 1 ครั้ง
	•	จำกัดเพดาน เช่น 8 หรือ 10 แก้วต่อคน

ตัวอย่าง rule
	•	lastDrinkAt ต้องห่างจากปัจจุบันอย่างน้อย 3 นาที
	•	ถ้าเกิน max แล้ว กดเพิ่มไม่ได้
	•	ถ้า checkout แล้ว กดไม่ได้

แบบนี้เบา dev และไม่เพิ่มภาระ staff

⸻

4) Flow เช็คเอาท์

สถานะ C2 — จบกิจกรรม

เมื่อผู้ใช้กด “เช็คเอาท์”
	1.	เปิด LIFF หน้า Check-out
	2.	ระบบสรุป:
	•	จำนวนแก้วทั้งหมด
	•	level ล่าสุด
	•	เวลาอยู่ในงาน
	3.	ผู้ใช้กดยืนยันออกจากงาน
	4.	ระบบบันทึก checkedOutAt
	5.	เปลี่ยน rich menu เป็นเมนูปิดงาน หรือกลับ default menu

เมนูหลังเช็คเอาท์
	•	ดูสรุปของฉัน
	•	รูปงาน / ข่าวสาร
	•	ติดต่อแอดมิน

⸻

5) โครงสร้าง Rich Menu ที่แนะนำ

LINE มีทั้ง default rich menu และ per-user rich menu
โดย default ตั้งได้ผ่าน LINE OA Manager หรือ Messaging API ส่วน per-user rich menu ต้องใช้ Messaging API และจะมี priority สูงกว่า default  ￼

ดังนั้นโครงที่เหมาะสุดคือ:

ใช้แบบผสม
	•	Default rich menu ทำใน LINE OA Manager
	•	Dynamic rich menu ตามสถานะ ทำด้วย Messaging API

⸻

ชุด Rich Menu ที่ควรมี

Menu 1: Guest

สำหรับคนที่ยังไม่ลงทะเบียน

ปุ่ม:
	•	ลงทะเบียน
	•	ตรวจสอบสถานะ
	•	สถานที่จัดงาน

⸻

Menu 2: Pending

สำหรับคนที่ลงทะเบียนแล้วแต่ยังไม่ผ่านอนุมัติ

ปุ่ม:
	•	ตรวจสอบสถานะ
	•	สถานที่จัดงาน
	•	ติดต่อแอดมิน

⸻

Menu 3: Confirmed

สำหรับคนที่คอนเฟิร์มแล้ว รอเข้างาน

ปุ่ม:
	•	เช็คอิน
	•	สถานที่จัดงาน
	•	กติกางาน

⸻

Menu 4: Checked-in

สำหรับคนที่เช็คอินแล้ว — Rich Menu รูปเดียวแนวนอน (เช่น 1200×405) สามโซนแตะ **ซ้าย → กลาง → ขวา** ตรงกับลำดับปุ่มบนภาพ

ปุ่ม:
	•	เติมเบียร์
	•	ระดับพลังของฉัน
	•	เช็คเอาท์

⸻

Menu 5: Completed

สำหรับคนที่เช็คเอาท์แล้ว

ปุ่ม:
	•	สรุปของฉัน
	•	รูปบรรยากาศ / ข่าวสาร
	•	ติดต่อแอดมิน

⸻

6) Mapping สถานะระบบ ↔ Rich Menu
สถานะระบบ                           Rich Menu
ยังไม่ลงทะเบียน              ->          Guest
ลงทะเบียนแล้ว / รอแอดมิน     ->          Pending
คอนเฟิร์มแล้ว                ->          Confirmed
เช็คอินแล้ว                  ->          Checked-in
เช็คเอาท์แล้ว                ->          Completed

ทุกครั้งที่ status เปลี่ยน backend จะเรียก Messaging API เพื่อ link rich menu ใหม่ให้ user คนนั้นผ่าน userId ได้โดยตรง  ￼

8) ลำดับเหตุการณ์จริงตั้งแต่ต้นจนจบ

ก่อนงาน
	1.	User add OA
	2.	เห็น Guest menu
	3.	ลงทะเบียน
	4.	ระบบเปลี่ยนเป็น Pending menu
	5.	Admin confirm
	6.	ระบบส่ง LINE แจ้งผล
	7.	ระบบเปลี่ยนเป็น Confirmed menu

ตอนมาถึงงาน
	8.	User กดเช็คอิน
	9.	LIFF ขอ location
	10.	ระบบตรวจรัศมี + เวลา
	11.	เช็คอินสำเร็จ
	12.	ระบบเปลี่ยนเป็น Checked-in menu

ระหว่างงาน
	13.	User กดเติมเบียร์
	14.	ระบบนับแก้ว
	15.	ระบบคำนวณ level
	16.	ถ้าแต้มถึง → level up

ตอนกลับ
	17.	User กดเช็คเอาท์
	18.	ระบบสรุปผล
	19.	ระบบเปลี่ยนเป็น Completed menu

9) จุดเด่นของ flow นี้

ข้อดี
	•	ไม่ต้องมี staff มานั่งสแกนเช็คอินทีละคน
	•	ผู้ใช้รู้เองว่าตอนนี้ต้องกดอะไรจาก rich menu
	•	เกมเก็บเลเวลช่วยให้งานมีสีสัน
	•	ระบบไม่ซับซ้อนเกินไปสำหรับการทำเร็ว
	•	ใช้ LINE OA เป็นศูนย์กลางเดียว

ข้อควรระวัง
	•	location อาจคลาดเคลื่อนได้ ควรมี fallback admin
	•	ปุ่มเติมเบียร์ต้องมี cooldown กันกดมั่ว
	•	ควรทำ wording ให้สนุก แต่ไม่ดูส่งเสริมการดื่มหนักเกินไป


สรุปสุดท้ายแบบสั้น

ระบบนี้จะทำงานแบบ:

ลงทะเบียน → รอคอนเฟิร์ม → เช็คอินด้วย location → ปลดล็อกเมนูเติมเบียร์ → เก็บเลเวลจากจำนวนแก้ว → เช็คเอาท์

และใช้ default rich menu เป็นเมนูตั้งต้น ส่วนเมนูที่เปลี่ยนตามสถานะให้ใช้ Messaging API แบบ per-user rich menu เพราะเหมาะกับ flow งานนี้ที่สุด
